# WebSocket

## What is WebSocket?

WebSocket is a **full-duplex**, **persistent**, **stateful** communication protocol that enables real-time, bidirectional communication between a client and a server over a **single TCP connection**.

Unlike HTTP, where communication follows a **request → response** model, WebSocket allows **either the client or the server to send messages at any time** once the connection is established.

**RFC:** 6455

---

# Why Do We Need WebSocket?

HTTP is excellent for request-response communication but is inefficient for applications requiring real-time updates.

Examples:

- Chat applications
- Live notifications
- Multiplayer games
- Stock market updates
- Sports score updates
- Collaborative editors (Google Docs)
- Live dashboards
- Online auctions

Without WebSocket, clients would have to repeatedly poll the server.

```text
Client
   │
GET /notifications
   │
No updates
   │
GET /notifications
   │
No updates
   │
GET /notifications
   │
New notification
```

This wastes:

- CPU
- Network bandwidth
- Database queries
- Server resources

---

# HTTP Polling vs Long Polling vs WebSocket

## Polling

```text
Client                 Server

GET /messages  ───────►

               ◄────── No new messages

(wait 5 sec)

GET /messages  ───────►

               ◄────── New messages
```

Problems:

- High latency
- Many unnecessary requests
- High server load

---

## Long Polling

Client sends request.

Server keeps the request open until data is available.

```text
Client                 Server

GET /messages ───────►

(wait...)

               ◄────── New message

GET /messages ───────►

(wait...)
```

Better than polling, but still creates many HTTP requests.

---

## WebSocket

Single connection.

```text
Client                 Server

Handshake

═══════════════════════

Client ◄────────────► Server

Client ◄────────────► Server

Client ◄────────────► Server
```

No repeated HTTP requests.

Very low latency.

---

# Characteristics

- Full Duplex
- Persistent Connection
- Low Latency
- Stateful
- Message-Oriented
- Runs over TCP
- Minimal protocol overhead

# WebSocket Handshake

A WebSocket connection begins as a normal HTTP request.

Client sends:

```http
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

Server responds:

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

After the `101 Switching Protocols` response, HTTP is no longer used.

The connection becomes a WebSocket connection.

---

# Important Handshake Headers

## Upgrade

Requests protocol upgrade.

```text
Upgrade: websocket
```

---

## Connection

Indicates the Upgrade header should be processed.

```text
Connection: Upgrade
```

---

## Sec-WebSocket-Key

Random Base64 value generated by the client.

Used to verify that the server understands the WebSocket protocol.

---

## Sec-WebSocket-Accept

Generated by the server from the client's key using SHA-1 and Base64 encoding.

Helps prevent accidental protocol confusion.

---

## Sec-WebSocket-Version

Current standardized version.

```text
13
```

```text
Client
    │
HTTP Upgrade Request
    │
101 Switching Protocols
    │
Connection Established
    │
Exchange Messages
    │
Close Frame
    │
TCP Connection Closed
```

# Message Flow

```text
Client

Hello
      ─────────►

          Server

      ◄────────

Hi!

      ─────────►

How are you?

      ◄────────

I'm good.
```

Both sides can send messages independently.

# WebSocket Frames

Communication happens using **frames**.

Common frame types:

| Opcode | Frame Type |
|---------|------------|
| 0x1 | Text |
| 0x2 | Binary |
| 0x8 | Close |
| 0x9 | Ping |
| 0xA | Pong |

---

# Text Frame

```text
Hello Server
```

UTF-8 encoded text.

---

# Binary Frame

Used for:

- Images
- Audio
- Video
- Protocol Buffers
- Custom binary protocols

---

# Ping/Pong Frames

Used to keep the connection alive.

Server:

```text
PING
```

Client:

```text
PONG
```

If Pong is not received within a timeout, the connection is typically considered dead and closed.

---

# Close Frame

Gracefully terminates the connection.

Example close codes:

| Code | Meaning |
|-------|---------|
| 1000 | Normal Closure |
| 1001 | Going Away |
| 1002 | Protocol Error |
| 1003 | Unsupported Data |
| 1008 | Policy Violation |
| 1009 | Message Too Big |
| 1011 | Internal Error |

---

# Fragmentation

Large messages may be split across multiple frames.

```text
Message

↓

Frame 1

↓

Frame 2

↓

Frame 3

↓

Reassembled Message
```

This allows streaming of large payloads without buffering the entire message first.

---

# Masking

All **client → server** frames are masked.

Server → Client frames are **not** masked.

Masking helps prevent certain attacks involving intermediaries (such as caches or proxies) misinterpreting WebSocket traffic.

---

# WebSocket States

```text
CONNECTING

OPEN

CLOSING

```

# Connection Management

Servers often maintain a connection object for each client.

```text
Connection

Socket ID

User ID

Authentication

Subscriptions

Heartbeat Timestamp
```

Large systems typically distribute connections across multiple WebSocket servers.

---

# Authentication

Authentication usually happens during the HTTP handshake.

Examples:

```text
Authorization: Bearer <JWT>
```

or

```text
Cookie: sessionId=...
```

The server validates credentials before accepting the upgrade.

---

# Broadcasting

One message sent to many clients.

Example:

```text
Chat Server

        │

 ┌──────┼──────┐

User A User B User C
```

Used in:

- Chat rooms
- Live scores
- Notifications
- Stock prices

---

# Rooms / Channels

Clients can subscribe to logical groups.

```text
Sports

Users:
A
B
C
```

```text
Finance

Users:
D
E
```

Server broadcasts only to subscribed clients.

---

# Scaling WebSocket Servers

One server can only maintain a finite number of connections.

Typical architecture:

```text
Clients

      │

Load Balancer

      │

 ┌──────────────┐
 │ WS Server 1  │
 │ WS Server 2  │
 │ WS Server 3  │
 └──────────────┘

      │

Redis Pub/Sub
Kafka
NATS

      │

Application Services
```

Pub/Sub systems ensure messages reach clients connected to different WebSocket servers.

---

# Advantages

- Very low latency
- Full-duplex communication
- Persistent connection
- Less protocol overhead than repeated HTTP requests
- Ideal for real-time applications

---

# Limitations

- Long-lived connections consume server memory and file descriptors.
- Requires heartbeat mechanisms to detect dead connections.
- Load balancing and scaling are more complex.
- Not ideal for simple request-response APIs.
- Some corporate proxies or firewalls may restrict WebSocket traffic.

---

# WebSocket vs HTTP

| Feature | HTTP | WebSocket |
|---------|------|-----------|
| Communication | Request-Response | Bidirectional |
| Connection | Usually short-lived | Persistent |
| Server Push | No (except via SSE/HTTP2 Push) | Yes |
| Latency | Higher | Very Low |
| Protocol Overhead | Higher | Lower |
| Real-Time Support | Limited | Excellent |

---

# WebSocket vs Server-Sent Events (SSE)

| Feature | WebSocket | SSE |
|---------|-----------|-----|
| Direction | Bidirectional | Server → Client |
| Transport | TCP | HTTP |
| Binary Support | Yes | No |
| Browser Support | Excellent | Excellent |
| Reconnection | Application-managed | Built-in by browser |
| Best Use Case | Chat, Games | Notifications, Live Feeds |
